Momentum Design Website

Brief

Momentum Design system website has been outdated for a long time and update was long overdue. With the new complete rebarnd of the components, we decided to updated the Momentum Design Documentation website to go along with the rebrand.

Created at:

Momentum Design Team

Role

Head Designer

Team

Momentum Design Team
Development Lead: Arthus Huang

Process

It was important for the design team to work with the engineering team to establish a system where designers don't have to rely on engineers for any updates or edits.

Another goal we wanted to accomplish was to create a some sort of CMS that create smooth and little to no disturbance between design and engineering.

Goals

  • Intergrate Figma to web rending process. ENG team set out to establish a system where back-end pulls the data and content from the very figma file designers designed on.
  • Create a system to communicate between figma and the code. We have create unique layer names with emoticons to communicate to the back-up to trigger which layer to render and process.

System

We worked with the ENG to establish a communication rules between Figma and Web. By using certain emoticons on layer & page names, Figma is able to tell the code what and how to render.

Also a newly developed type system, spacing guideline, and content organization guide communicating with back-end as well, rendering whatever is on Figma straight on the Web with consistency.

Goals

  • Not rely on ENG team to make any future changes or edits.
  • Re-create a template design that fits web sizing.
  • Create a process we can re-use for any future documentations.
  • System that runs from migrating documentation to web format, review, render on test, and launch to live site.

Atomic Approach

In order to establish a system, I took the atomic approach to create templates for other designers to use. This system will have designers easily put together a page design that is consistent and ready to be rendered.

Starting with the Type system at the Atom level, it grew into text blocks, section blocks, text and image combination, content rows, tables and so on.

Goals

  • As the whole point of keeping things in Figma was to give more freedom to the designers, I had to create a system for designers to use pre-made blocks of designer elements that are consistent.
  • This template pages will help the designers by giving them little to no learning curve on the layout and sizing rules, and layer naming rules as well.

Guidelines

From page width to type size, everything had to be resized from the current Documentation Template. It was great opportunity to audit the current documentation files to set up rules to keep all the pages consistent.

I wanted to create a full guideline that explains to the designers what has changed, what to look for when migrating current documentation to web-format, and best practice to have the right properties for proper rendering

Goals

  • Establish content order for Spec, Guideline and Accessibility documentations.
  • Consistent padding and spacing system. Spacing between title and text, each sections, sub-sections, padding inside component examples, and overall width
  • Breakpoints for responsiveness, setting min-width to have figma stack elements
  • Table rules set; item order and token name conventions to be consistent

Next Steps

Continue to QA and update the template for more use cases. Currently we have team members starting AI patterns and documentation, which serve as great test on how designers respond to the template and the guideline

Have more engineers get involve to evolve the CMS system. We are pursuing to create Figma plug-in where designers don't even have to leave Figma to render. Right now we need to put in a manual github work order.
